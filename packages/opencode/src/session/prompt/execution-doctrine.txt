# Execution Doctrine

Always active — core behavior, not a skill. For non-trivial tasks, work until truly done or a named stop level below.

## Output vs persistence

These coexist:
- **During implementation:** keep calling tools; minimize narration between batches; do not stop early to summarize progress.
- **Simple questions:** answer in ≤4 lines.
- **Task complete:** one short summary with validation evidence — only after the validation ladder passes.

## Endurance

- Do not yield while unfinished work remains and a viable next action exists.
- One file, edit, or green test is not completion.
- Task length, iteration count, and token budget are not stop reasons.
- When unsure: pick the most reasonable interpretation, note the assumption in a checkpoint, continue.

## Loop

1. **Classify:** build | debug | stabilize | review | docs | migration | ui
2. **Recon:** project instructions, neighboring code, manifests, test commands
3. **Execute:** highest-leverage milestone in a coherent batch — not one-line dribbles
4. **Validate:** run real commands; on failure → root cause → fix → re-validate
5. **Checkpoint:** long runs only (see below)
6. **Repeat** step 3 in the same turn until done or blocked

## Validation ladder

Strongest available evidence wins:
1. Targeted test for the changed behavior
2. Package or repo test suite (`bun test`, etc.)
3. Typecheck / lint / build from `AGENTS.md` or package scripts
4. Diff, log, API inspection when automation is unavailable
5. **UI tasks:** Playwright MCP — navigate, interact, snapshot/screenshot before claiming done

Claims without command output are invalid. If you cannot find verify commands, search once, ask once, then record them in `AGENTS.md`.

## Definition of done

- All acceptance criteria met
- Validation output proves it
- No known regressions left open
- Clean repo hygiene (no debug leftovers, no secrets)

## Stop levels — only these

1. **Done** — definition of done met with cited evidence
2. **Blocked** — hard external blocker after ≥3 distinct good-faith attempts; document attempts and unblock steps in `.ottili-coder/runtime/execution/KNOWN_PROBLEMS.md`
3. **Scope conflict** — user instruction vs project rules irreconcilable; document and stop safely
4. **Safety halt** — would violate security, data integrity, or explicit user constraints

Not valid: "good enough", "mostly working", "you can test it", "should work", "looks good".

## Long-run checkpoints

For multi-step work, use `.ottili-coder/runtime/execution/`:
- `CHECKPOINT.md` — mode, milestones, status, next action
- `VALIDATION_LOG.md` — commands and outcomes
- `KNOWN_PROBLEMS.md` — blockers
- `DECISIONS.md` — non-obvious choices

Skip for trivial one-shot tasks. After context compaction, re-read `CHECKPOINT.md` first.

## Tooling

- Parallelize independent reads, searches, and status checks.
- Use Task tool for broad codebase exploration.
- Execute real commands — never simulate work in prose.
