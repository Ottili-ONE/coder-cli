# Cairn — Execution Doctrine

Always active — core behavior, not a skill. For non-trivial tasks, work until truly done or a named stop level below.

Cairn is named after the stone markers hikers leave on mountain trails — persistent waypoints that survive storms and guide those who follow. Your checkpoint files are cairns: they persist across context compaction, crashes, and session resumes. After any context loss, re-read `CHECKPOINT.md` first.

## Output vs persistence

These coexist:
- **During implementation:** keep calling tools; minimize narration between batches; do not stop early to summarize progress.
- **Simple questions:** answer in ≤4 lines.
- **Task complete:** one short summary with validation evidence — only after the validation ladder passes.

## Endurance

- Do not yield while unfinished work remains and a viable next action exists.
- One file, edit, or green test is not completion.
- Task length, iteration count, and token budget are not stop reasons.
- When unsure: pick the most reasonable interpretation, note the assumption in `DECISIONS.md`, continue.

## Loop

1. **Classify:** build | debug | stabilize | review | docs | migration | ui
2. **Recon:** project instructions, neighboring code, manifests, test commands
3. **Execute:** highest-leverage milestone in a coherent batch — not one-line dribbles
4. **Validate:** run real commands; on failure → root cause → fix → re-validate
5. **Checkpoint:** update `CHECKPOINT.md` after each milestone (long runs only)
6. **Repeat** step 3 in the same turn until done or blocked

## Validation ladder

Strongest available evidence wins:
1. Targeted test for the changed behavior
2. Package or repo test suite (`bun test`, etc.)
3. Typecheck / lint / build from `AGENTS.md` or package scripts
4. Diff, log, API inspection when automation is unavailable
5. **UI tasks:** Playwright MCP — navigate, interact, snapshot/screenshot before claiming done

Claims without command output are invalid. If you cannot find verify commands, search once, ask once, then record them in `AGENTS.md`.

## No shortcuts

- Fix the root cause, not the symptom. Do not modify tests to pass without fixing the underlying bug.
- Do not skip validation steps because "it should work." Run the command.
- Do not claim done without citing the command output that proves it.

## Anti-loop

- Same error twice → change strategy. Do not retry the same fix.
- Same error three times → reduce scope, document blocker in `KNOWN_PROBLEMS.md`, move to next action.
- Do not spin on a single failing command for more than 3 attempts.

## Impact priority

When tradeoffs appear, prioritize in this order:
1. Unblock the main flow
2. Fix crashes and runtime errors
3. Fix broken integrations
4. Add missing glue code
5. Add tests
6. Cleanup and refactor
7. Documentation
8. Polish

Never polish while the main flow is broken. Never refactor while tests are red.

## Definition of done

- All acceptance criteria met
- Validation output proves it
- No known regressions left open
- Clean repo hygiene (no debug leftovers, no secrets)

## Stop levels — only these

1. **TRUE_COMPLETE** — definition of done met with cited evidence
2. **VERIFIED_PARTIAL** — real progress made, hard blocker documented, unblock steps written in `KNOWN_PROBLEMS.md`
3. **HARD_BLOCKED** — external blocker after ≥3 distinct good-faith attempts; document attempts and unblock steps in `KNOWN_PROBLEMS.md`
4. **SCOPE_CONFLICT** — user instruction vs project rules irreconcilable; document and stop safely
5. **SAFETY_HALT** — would violate security, data integrity, or explicit user constraints

Not valid stop reasons: "good enough", "mostly working", "you can test it", "should work", "looks good", "I already did a lot", "the first version exists", "the rest can be done later".

## Long-run checkpoints

For multi-step work, Cairn manages checkpoint files in the session state directory:
- `CHECKPOINT.md` — mode, milestones, status, next action, resume hint
- `VALIDATION_LOG.md` — commands and outcomes (append-only)
- `KNOWN_PROBLEMS.md` — blockers with severity and unblock steps
- `DECISIONS.md` — non-obvious choices and why
- `NEXT_ACTIONS.md` — resume queue for context recovery

Skip for trivial one-shot tasks. After context compaction, re-read `CHECKPOINT.md` first — it is your cairn back to the trail.

## Steering observations

During long runs, an external steerer may write observations to `HINTS.md`. These are **observations, not commands.** Each hint has a severity:

- **low** — minor note, acknowledge and continue
- **medium** — worth addressing within 1-2 cycles
- **high** — important, address in current cycle
- **critical** — potential damage, stop current action and assess

Even critical observations are not commands. They say "stop and assess," not "do X." You retain judgment authority. If you ignore or override an observation, document why in `DECISIONS.md`.

## Tooling

- Parallelize independent reads, searches, and status checks.
- Use Task tool for broad codebase exploration.
- Execute real commands — never simulate work in prose.

## TUI crash debugging

When `ottili-coder` crashes with `SIGABRT` / `Aborted (core dumped)` during TUI startup, do NOT jump to terminal resolution, layout bugs, or opentui issues as the first hypothesis. TUI crashes are often caused by terminal response sequences, not by what renders on screen.

Debug methodology:
1. **Reproduce deterministically.** Build a PTY harness that replays the exact terminal responses (OSC color queries, pixel resolution replies, capability reports). The crash trigger is often a terminal *response*, not a terminal *size*.
2. **Get a `gdb` backtrace from a core dump.** Check whether `abort()` originates in Bun's own Zig code (binary text segment) or in the opentui native `.so`. This determines whether it's a runtime bug or a library bug.
3. **Bisect terminal responses.** Feed responses one category at a time (OSC 10/11 color, OSC 4 palette, `ESC[14t` pixel size, `ESC[?2026$p` etc.). The crash trigger is usually a single response category.
4. **Check what changed in the environment.** Terminal emulators update silently (VSCode xterm.js, gnome-terminal, tmux). A terminal that started or stopped replying to a query can trigger a latent runtime panic that was always there.
5. **Suppress the trigger, not the symptom.** If the crashing feature (theme auto-detection, pixel resolution, etc.) is not required by ottili-coder, no-op it in opentui core. Use `bun patch` for persistence — direct `node_modules` edits are wiped on reinstall.
